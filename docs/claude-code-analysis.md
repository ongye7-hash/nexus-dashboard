# Claude의 솔직한 분석

이건 자동화된 리뷰가 아닙니다. 코드를 직접 읽고 내가 느낀 것을 솔직하게 씁니다.

---

## 결론부터

**기능은 많다. 너무 많다.** 하루 만에 15,000줄이 넘는 코드가 만들어졌다. 23개 API, 25개 컴포넌트. 근데 이 중에서 **실제로 테스트된 기능은 "프로젝트 목록 보기"와 "새로고침" 정도**다.

나머지는 전부 "만들어져 있지만 실제로 돌려본 적 없는" 코드다.

---

## 1. 가장 큰 문제: 테스트 없이 쌓아올린 코드

오늘 하루 동안:
- 내장 터미널을 만들었지만 **실제로 명령어를 쳐본 적 없다**
- VPS SSH 연결을 구현했지만 **실제 서버에 연결해본 적 없다**
- Claude Code 파이프를 만들었지만 **실제로 리뷰가 적용되는지 확인 안 했다**
- GitHub 클론을 구현했지만 **실제로 레포를 클론해본 적 없다**
- AI 어시스턴트를 Claude API로 바꿨지만 **API 키를 넣고 분석을 돌려본 적 없다**

코드는 컴파일되고, 빌드는 성공한다. 하지만 **"빌드 성공 ≠ 동작"** 이다.

이 프로젝트에 지금 가장 필요한 건 새 기능이 아니라 **기존 기능을 실제로 써보는 것**이다.

## 2. 구조적 문제: MorningCodex가 너무 크다

```
MorningCodex.tsx  — 1,270줄, useState 16개, fetch 13번
ProjectModal.tsx  — 858줄, useState 17개
page.tsx          — 925줄, useState 16개
```

MorningCodex 하나가 1,270줄이다. 이 안에:
- 인사말
- 어제 요약
- 스트릭
- 알림
- 실행 중인 서버
- 작업 프로젝트 (git 정보 + 빠른 커밋 + Resume)
- 파일 변경 피드
- Morning Routine
- 작업 세션 타이머

이건 **9개의 섹션이 하나의 컴포넌트에 들어있는 것**이다. 각각이 독립적인 데이터 소스를 가지고, 독립적인 상태를 관리한다. 한 곳에서 에러가 나면 전체가 죽는다.

이건 "모놀리식 컴포넌트" 문제다. 나중에 하나만 수정하려 해도 1,270줄을 다 이해해야 한다.

## 3. 구조적 문제: database.ts가 신 클래스(God Class)다

```
database.ts — 797줄, 49개 함수, 11개 테이블
```

하나의 파일에서:
- 프로젝트 메타데이터
- 그룹
- 포트 매핑
- 작업 세션
- 일일 활동
- 사용자 통계
- 뱃지
- GitHub 레포
- VPS 서버
- TODO
- 설정

전부 관리한다. 관계 없는 11개 도메인이 하나의 파일에 있다. 함수 하나 찾으려면 800줄을 스크롤해야 한다.

## 4. 사용자 경험 문제: 설정이 3곳에 분산되어 있다

- **GitHub 설정** → 사이드바 "설정" 버튼
- **VPS 설정** → 사이드바 "VPS 설정" 버튼
- **AI API 키** → 프로젝트 모달 → AI 어시스턴트 패널 안

사용자가 처음 쓸 때 "API 키를 어디서 넣지?" 하고 찾아야 한다. **한 곳에서 전부 설정할 수 있어야 한다.**

---

## 5. 반복 패턴 문제: 보안 수정을 4번이나 했다

오늘의 커밋을 보면:

1. GitHub 구현 → 코드 리뷰 → 보안 이슈 5건 수정
2. VPS 구현 → 코드 리뷰 → 보안 이슈 5건 수정
3. AI 개선 → 코드 리뷰 → 보안 이슈 5건 수정
4. 심층 감사 → 보안 이슈 20건 수정

왜 매번 같은 종류의 문제가 반복됐나?
- `execSync`에 사용자 입력 직접 삽입 → **매번 발생**
- `validateProjectPath` 누락 → **매번 발생**
- 토큰/키 평문 저장 → **매번 발생**

이건 "보안 체크리스트"가 없기 때문이다. 새 API를 만들 때마다 같은 실수를 반복하고, 리뷰에서 잡고, 수정하는 사이클.

## 6. 현실적 우려: 이 프로젝트의 유지보수

15,000줄의 코드를 혼자서 유지보수하려면:
- **테스트가 없다** → 뭔가 바꾸면 다른 게 깨졌는지 모른다
- **문서화가 부족하다** → API 스펙, 컴포넌트 props 문서가 없다 (보고서는 있지만 코드 레벨 문서가 없다)
- **의존성이 17개** → 이 중 node-pty, ssh2, better-sqlite3는 네이티브 모듈이라 Node.js 업그레이드 시 깨질 수 있다

---

## 7. 좋은 점 (솔직하게)

까다롭게 봤지만, 잘 된 부분도 있다:

- **기술 스택 선택이 좋다** — Next.js 16 + SQLite는 로컬 도구에 최적
- **보안 수정이 실제로 이루어졌다** — 대부분의 개인 프로젝트는 보안 리뷰 자체를 안 한다
- **AES-256-GCM 암호화** — 로컬 도구치고 과할 수 있지만, 올바른 방향
- **path-validator** — 별도 모듈로 분리된 경로 검증은 좋은 패턴
- **GitHub/VPS 연동 구조** — DB 스키마와 API 설계가 합리적
- **PWA 지원** — 로컬 도구를 앱처럼 쓸 수 있게 한 건 좋다

## 추가 분석: 아까 못 잡은 것들

### A. 프로젝트 ID가 매 요청마다 바뀐다

```typescript
// projects/route.ts:293
id: `proj-${Date.now()}-${Math.random().toString(36).slice(2, 11)}-${projects.length}`
```

프로젝트 ID가 `Date.now() + Math.random()`으로 생성된다. **페이지를 새로고침할 때마다 모든 프로젝트의 ID가 바뀐다.** 이게 왜 문제냐:
- React의 key가 매번 바뀌므로 모든 카드가 매번 처음부터 다시 렌더링된다
- 프로젝트 ID 기반의 캐시, 상태 참조가 새로고침 시 전부 무효화된다
- `gitInfoMap[project.id]` — 새로고침하면 이전에 가져온 git 정보가 전부 날아간다

프로젝트 경로의 해시값 같은 **결정론적 ID**를 써야 한다.

### B. 프로젝트 스캔이 동기적이고 느리다

`projects/route.ts` GET 핸들러가 하는 일:
1. `fs.readdirSync` — Desktop 전체 읽기
2. 각 폴더마다 `fs.existsSync` 11번 (프로젝트 식별)
3. 각 폴더마다 `detectProjectType` — package.json 읽기 + 파싱
4. 각 폴더마다 `getDirectorySize` — 재귀 탐색 (2단계)
5. 각 폴더마다 `detectDeployUrl` — vercel.json, package.json 읽기
6. 각 폴더마다 `execSync('git remote get-url origin')` — **30개 프로젝트면 30번 git 명령 실행**

이 전부가 **동기적**으로 실행된다. 프로젝트가 30개면 수 초가 걸린다. 이게 무한로딩처럼 느껴졌던 원인이기도 하다.

### C. `execGit` 함수가 여전히 쉘 인터폴레이션을 쓴다

```typescript
// git/route.ts:44
return execSync(`git ${command}`, { ... });
```

git/route.ts의 GET 핸들러에서 쓰는 `execGit` 함수는 여전히 `execSync`에 문자열을 직접 넣는다. POST의 commit은 `execFileSync`로 수정했지만, **GET에서 쓰는 `execGit`은 안 바꿨다.** `command` 파라미터가 항상 하드코딩된 git 명령이라 실질적 위험은 낮지만, 패턴이 일관되지 않다.

### D. insights/route.ts에서 branch 이름이 쉘에 들어간다

```typescript
// insights/route.ts:54
execSync(`git rev-list --left-right --count ${branch}...origin/${branch}`, ...)
```

`branch`는 `git rev-parse --abbrev-ref HEAD`의 출력인데, 이론적으로 브랜치 이름에 쉘 메타문자가 있을 수 있다 (예: `feat/$(calc)`). 가능성은 낮지만 패턴이 위험하다.

### E. 빈 catch 블록이 40개

```
grep "catch {" src/app/api/ → 40건
```

에러를 조용히 삼키는 빈 catch가 40개다. 이 중 상당수는 합리적이지만 (선택적 기능의 fallback), 디버깅할 때 "왜 안 되지?" 하고 한참 찾게 만드는 원인이 된다. 최소한 `console.warn`이라도 넣어야 한다.

### F. MorningCodex의 데이터 로딩이 waterfall이다

```
페이지 로드
  → fetchProjects (page.tsx)
    → MorningCodex 렌더
      → fetchGitInfo (각 프로젝트마다 /api/git 호출)
      → fetchProcesses
      → fetchStats
      → fetchActiveSessions
```

MorningCodex가 마운트된 후에야 4개의 API를 호출한다. 그리고 `fetchGitInfo`는 pinned 프로젝트 6개에 대해 **각각 개별 fetch**를 한다. 즉 페이지 로드 시 최소 **7~10개의 API 요청이 순차적으로 또는 반병렬적으로** 발생한다.

이걸 하나의 `/api/dashboard` 엔드포인트로 합치면 1번의 요청으로 줄일 수 있다.

### G. 나 자신에 대한 비판

솔직히 오늘 내가 만든 코드에 대해서도 비판해야 한다:

1. **내가 보안 실수를 4번 반복했다** — 첫 구현에서 `execSync`에 직접 입력을 넣은 건 내 실수다. "보안 체크리스트가 없었다"고 했지만, 나한테 체크리스트가 내장되어 있어야 한다.

2. **기능을 너무 빠르게 추가했다** — 사용자가 "전부 다 해"라고 했을 때, "이것만 먼저 하고 테스트하자"라고 했어야 했다. 대신 3개 에이전트를 병렬로 돌려서 한번에 구현했다. 속도는 빨랐지만 품질이 희생됐다.

3. **에이전트에게 너무 많이 위임했다** — MorningCodex 1,270줄을 에이전트가 한 번에 작성했다. 내가 한 줄 한 줄 검토하지 않았다. 결과적으로 모놀리식 컴포넌트가 만들어졌다.

## 10. 최종 평가

**이 프로젝트는 "프로토타입이 끝난 단계"다.**

기능은 프로덕트 수준으로 많지만, 코드 품질은 프로토타입 수준이다. 이건 나쁜 게 아니다 — 하루 만에 이만큼 만든 건 대단하다. 하지만 **이걸 "제품"으로 만들려면 지금부터는 기능 추가가 아니라 안정화에 집중**해야 한다.

프로토타입 → 제품 전환에 필요한 건:
1. 실사용 (2주)
2. 구조 정리 (MorningCodex, database.ts)
3. 설정 통합
4. 기본 테스트
5. 그 후에야 새 기능

**한 줄 요약: 만드는 건 끝났다. 이제 써라.**
