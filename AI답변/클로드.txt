1. 기술적 약점 — 공격 가능 지점
Next.js로 로컬 툴을 만든 건 오버엔지니어링입니다. 웹 프레임워크로 로컬 파일시스템을 건드리는 구조는 태생적으로 보안 제약과 권한 문제를 안고 갑니다. API Route를 통해 child_process.exec으로 VSCode나 터미널을 여는 구조일 텐데, 이건 사실상 로컬에 RCE(Remote Code Execution) 엔드포인트를 열어놓는 겁니다. 저라면 여기를 공격합니다 — "Nexus는 보안 모델이 없다"고.
데이터베이스가 없습니다. 메모, 태그, 상태, 그룹 같은 사용자 데이터를 어디에 저장하는지 브리프에 명시가 없어요. JSON 파일이겠죠. 이건 동시성 문제, 데이터 유실, 마이그레이션 불가를 의미합니다. SQLite 하나만 넣어도 되는 걸 안 한 건 약점입니다.
스캐너의 확장성 한계. "바탕화면 폴더 스캔"이라는 접근은 프로젝트가 50개 넘어가면 느려지고, monorepo는 인식 못 하고, Docker/Rust/Go/Java 같은 생태계는 빠져 있을 가능성이 높습니다. 저라면 더 많은 언어와 빌드 시스템을 지원하는 스캐너를 만들어서 "Nexus가 못 잡는 프로젝트도 우리는 잡는다"고 포지셔닝합니다.
Electron이 아닌 브라우저 기반이라는 한계. PWA로 커버하려 하지만, 시스템 트레이 상주, 파일 워처(실시간 변경 감지), 글로벌 단축키, 시스템 알림 같은 네이티브 기능은 브라우저에서 불가능합니다. "웹앱인데 로컬 툴이라고 주장하는" 어정쩡한 포지션입니다.

2. 놓치고 있는 사용자 니즈
팀/협업은 완전히 무시하고 있습니다. 1인 개발자만 타겟인데, 현실에서 "프로젝트가 흩어져 있다"는 고통은 팀에서 더 큽니다. 온보딩할 때 신규 팀원이 "이 프로젝트 어디서 클론받아요? 어떻게 실행해요?"라고 물어보는 상황이요. 여기를 완전히 놓치고 있습니다.
프로젝트의 "건강 상태"에 대한 니즈. 개발자가 진짜 알고 싶은 건 "이 프로젝트 의존성 얼마나 오래됐지? 보안 취약점 있나? node_modules 얼마나 먹고 있지? 마지막 빌드 성공했나?"입니다. Nexus는 예쁜 카드만 보여주고, 이런 실질적 헬스체크는 없습니다.
"왜 이 프로젝트를 중단했지?"를 기억하는 기능이 없습니다. 개발자가 6개월 전 프로젝트를 다시 열었을 때 필요한 건 파일 트리가 아니라 "당시 어디까지 했고, 왜 멈췄고, 다음에 뭘 해야 하는지"입니다. 메모 기능이 있지만, 능동적으로 컨텍스트를 복원해주는 건 아닙니다.
디스크 공간 관리. node_modules 폴더 하나가 수백 MB입니다. 안 쓰는 프로젝트 30개의 node_modules가 10GB 넘게 먹고 있는 상황은 흔합니다. "안 쓰는 프로젝트 정리해서 50GB 확보하세요" 같은 기능은 즉각적인 가치를 줍니다. 이걸 안 하고 있습니다.

3. 내가 똑같은 아이디어로 경쟁한다면
첫째, Electron/Tauri로 데스크톱 네이티브로 갑니다. 시스템 트레이에 상주하면서 파일 변경을 실시간 워치하고, 글로벌 단축키(Spotlight/Raycast 스타일)로 어디서든 프로젝트를 검색해서 바로 여는 겁니다. "브라우저 탭 하나 더 여는 게 아니라, OS에 녹아드는 도구"로 차별화합니다.
둘째, 포트폴리오 대신 "개발자 타임머신" 방향으로 갑니다. 포트폴리오 생성기는 레드오션입니다. 대신, Git 히스토리 + 파일 변경 + 작업 시간을 자동 추적해서 "당신이 지난 1년간 뭘 만들었는지" 보여주는 개인 분석 도구로 포지셔닝합니다. 연말 회고, 성과 평가, 이력서 작성 시 실제 데이터 기반으로 "나는 올해 12개 프로젝트에서 847커밋을 했고, 주로 TypeScript와 Python을 썼다"를 뽑아주는 겁니다. 이건 경쟁자가 거의 없습니다.
셋째, CLI-first로 만듭니다. nx open my-project, nx list --stale, nx clean node_modules --all 같은 CLI를 먼저 만들고, 웹 대시보드는 그 위의 UI 레이어로 둡니다. 개발자는 터미널에 살아요. 브라우저를 열어야 쓸 수 있는 도구는 습관이 안 됩니다.

시나리오 2: Phase 3~6을 다 만들다가 포커스를 잃는다.

1. "진짜 좋다" vs "별로 안 쓸 것 같다"
진짜 좋다 — 매일 쓸 기능
커맨드 팔레트 (Ctrl+K). 이게 핵심입니다. 프로젝트 67개면 마우스로 스크롤해서 찾는 건 불가능해요. Ctrl+K → "pay" 입력 → 결제 프로젝트 선택 → Enter로 VSCode 열기까지 2초면 끝나는 플로우가 되면, 이것 하나만으로도 설치할 이유가 됩니다.
프로젝트 자동 감지 + 기술 스택 추출. 수동 등록이었으면 절대 안 씁니다. 폴더 지정만 하면 알아서 잡아주는 건 진입 장벽을 없애주는 결정적 기능이에요.
상태 관리 (개발중/활성/배포됨/보관됨). 67개 프로젝트 중 실제로 활발한 건 8~10개입니다. 나머지는 "보관됨"으로 치워놓고 활성 프로젝트만 보는 필터링, 이건 진짜 필요합니다.
VSCode/터미널/서버 원클릭 실행. 매일 아침 하는 루틴이 "어떤 폴더였지 → 탐색기 열고 → 우클릭 → Open with Code"인데, 이걸 한 번에 해결해주면 체감이 큽니다.
즐겨찾기/핀. 단순하지만 필수. 지금 집중하는 3~4개 프로젝트를 상단에 고정하는 것, 이거 없으면 매번 찾아야 해요.
별로 안 쓸 것 같다 — 솔직히
그리드 뷰 카드 UI. 예쁘긴 한데, 프로젝트가 20개 넘어가면 카드는 안 봅니다. 리스트 뷰로 고정하고 다시는 그리드로 안 갈 거예요. 개발 시간 많이 들었을 텐데, 실사용에서는 리스트가 압도적으로 빠릅니다.
통계 대시보드 (전체/활성/배포됨/이번주). 솔직히 "나한테 프로젝트가 47개 있다"는 정보는 한 번 보고 끝입니다. 매일 볼 이유가 없어요. 이 영역은 다른 걸로 채워야 합니다.
모바일 반응형. 모바일에서 로컬 개발 프로젝트를 관리하는 시나리오가 현실에 없습니다. 지하철에서 "아 저 프로젝트 상태를 바꿔야지" 하는 사람은 없어요. 이 시간을 데스크톱 경험에 쏟는 게 나았습니다.
파일 트리 미리보기. VSCode를 열면 더 좋은 파일 트리가 있는데, 대시보드에서 파일 트리를 왜 볼까요. 한 번도 안 쓸 것 같습니다.
라이브 프리뷰 (iframe). 서버가 꺼져 있으면 빈 화면이고, 켜져 있으면 이미 브라우저 탭에 열려 있습니다. 쓸 타이밍이 없어요.
PWA 지원. 이건 "있으면 좋지" 카테고리인데, 실제로 PWA로 설치해서 쓸 사람은 거의 없을 겁니다. 개발 리소스 대비 효과가 낮았을 거예요.

2. 첫 5분 안에 하고 싶은 작업
이 순서 그대로입니다:
0~30초: "내 프로젝트 폴더 3개를 스캔 소스로 등록" — 바탕화면만으로는 부족해요. ~/projects, ~/work, ~/side-projects 이 세 군데를 한 번에 지정하고 싶습니다.
30초~1분: 스캔 결과를 보면서 "이건 프로젝트가 아닌데"하는 것들(node_modules 안의 뭔가, 임시 폴더 등)을 빠르게 제외. .nexusignore 같은 게 있으면 좋겠고, 없으면 UI에서 "이건 숨기기" 클릭.
1~2분: 지금 작업 중인 프로젝트 3개를 핀 고정. 나머지 전부를 "보관됨"으로 일괄 변경. 하나하나 클릭하는 게 아니라 체크박스 선택 → 일괄 상태 변경이 가능해야 합니다.
2~3분: Ctrl+K 쳐보고 검색이 얼마나 빠른지 체감. "pay"를 치면 payment-gateway, payroll-app이 바로 나오는지. 여기서 느리거나 결과가 이상하면 바로 끕니다.
3~5분: 핀된 프로젝트 하나를 클릭해서 VSCode로 여는 플로우를 테스트. 클릭 → VSCode 열림까지 1초 이내여야 합니다. 2초 넘으면 "그냥 터미널에서 code . 치는 게 빠르네" 하고 끕니다.
핵심은, 첫 5분의 경험이 "기존 방식보다 확실히 빠르다"를 증명해야 합니다. 그렇지 않으면 6분째에 브라우저 탭을 닫습니다.

3. 매일 반복 루틴 자동화
제 아침 루틴이 이겁니다:
1. 슬랙 확인 → 오늘 뭘 해야 하는지 파악
2. 활성 프로젝트 3개의 git pull
3. 의존성 변경 있으면 npm install
4. dev 서버 실행
5. 브라우저에서 localhost 열기
이걸 Nexus가 해결하려면 "Morning Routine" 기능이 필요합니다:
프로젝트 그룹 일괄 실행. "출근" 버튼 하나를 누르면, 지정된 프로젝트 3개에 대해 git pull → npm install(변경 시만) → npm run dev를 순차 실행. 각 프로젝트의 서버 상태가 대시보드에 실시간으로 보여야 합니다 — "project-a: 서버 실행 중 :3000", "project-b: pull 중...", "project-c: 빌드 에러".
서버 상태 모니터링. 지금 몇 번 포트에서 뭐가 돌고 있는지를 대시보드에서 보여주세요. lsof -i 결과를 예쁘게 보여주는 것만으로도 가치가 있습니다. "아 3000번 포트 이미 쓰고 있네"를 찾으려고 터미널 치는 시간이 꽤 됩니다.
자동 git pull + 변경사항 요약. 아침에 대시보드를 열면 "어젯밤 동료가 이런 커밋을 했습니다"를 프로젝트별로 보여주면, 슬랙에서 PR 링크를 하나하나 클릭하는 것보다 빠릅니다.
스크립트 프리셋. 프로젝트마다 실행 명령이 다릅니다. 어떤 건 npm run dev, 어떤 건 docker-compose up, 어떤 건 python manage.py runserver. 프로젝트별로 커스텀 스크립트를 등록하고 한 방에 실행할 수 있어야 합니다.

4. 프로젝트 간 코드 재사용
이건 제가 가장 고통받는 부분입니다. "저번에 만든 인증 미들웨어 어디 있더라?"를 매번 겪습니다.
필요한 기능:
코드 스니펫 인덱싱. 프로젝트를 스캔할 때 주요 파일(utils, hooks, middleware, components)을 인덱싱해서, "useAuth"를 검색하면 어떤 프로젝트의 어떤 파일에 있는지 바로 나오게. 전체 코드를 인덱싱할 필요는 없고, export된 함수/컴포넌트 이름 수준이면 됩니다.
크로스 프로젝트 검색. 현재 커맨드 팔레트는 프로젝트 이름만 검색하는 것 같은데, Ctrl+K → "debounce" 치면 "project-a/utils/debounce.ts, project-c/hooks/useDebounce.ts"가 나와야 합니다. 이게 되면 게임 체인저입니다.
스니펫 라이브러리. 자주 쓰는 코드 조각을 태그 달아서 저장. "auth", "api-client", "error-handler" 같은 태그로 분류. 새 프로젝트 시작할 때 여기서 꺼내 쓰는 플로우.
의존성 비교 매트릭스. 프로젝트 A는 axios 0.21, 프로젝트 B는 axios 1.6을 쓰고 있다면, 이걸 한눈에 보여주세요. "이 10개 프로젝트가 전부 다른 버전의 같은 라이브러리를 쓰고 있다"는 정보가 가치 있습니다.

5. 프로젝트 열 때 바로 보고 싶은 정보
파일 트리와 README는 솔직히 5순위 밖입니다. 제가 프로젝트를 열었을 때 눈이 가는 순서:
1순위: 마지막 컨텍스트. "내가 마지막으로 이 프로젝트에서 뭘 했지?" — 마지막 커밋 메시지, 마지막으로 수정한 파일 3개, 내가 적어둔 TODO/메모. 이게 0.5초 안에 보여야 합니다.
2순위: 현재 브랜치 + uncommitted 변경사항. git status를 대시보드에서 바로 보고 싶어요. "아, 이 프로젝트 작업하다 말았네. feature/payment 브랜치에 수정 3개가 커밋 안 됐구나."
3순위: 서버/빌드 상태. 지금 서버가 돌고 있는지, 마지막 빌드가 성공했는지, 에러가 있는지.
4순위: 의존성 건강. outdated 패키지가 몇 개인지, 보안 취약점이 있는지. npm audit 결과 요약.
5순위: 연결된 외부 리소스. Vercel 배포 URL, GitHub 레포 링크, Figma 디자인 링크, API 문서 링크. 이걸 등록해놓으면 "이 프로젝트 관련 링크 어디 있지" 하고 북마크 뒤지는 시간이 사라집니다.

추가: 당신이 안 물어봤지만 물어봐야 할 것들
"누구를 위한 도구인가"를 다시 정의하세요
브리프를 보면 타겟이 모호합니다. 취준생? 프리랜서? 시니어 개발자? 전부 다른 사람들이에요. 제 관점에서, **"프로젝트 10개 이상을 동시에 관리하는 중급~시니어 개발자"**로 좁히세요. 프로젝트 3개인 주니어에게는 이 도구가 과합니다. 50개 넘는 시니어에게는 절실합니다.
포트폴리오 생성기 방향에 대한 제 반대 의견
이전 분석에서 방향 A(포트폴리오 생성기)를 추천했는데, 저는 반대입니다. 이유는 포트폴리오 생성기는 1년에 한두 번 쓰는 도구이고, 매일 쓰는 도구가 아니기 때문입니다. "매일 쓰는 개발 허브"라는 목표와 정면으로 충돌해요. 포트폴리오는 부가 기능으로 넣되, 핵심은 **"일상 개발 워크플로우의 시작점"**이어야 합니다.
진짜 경쟁상대는 Raycast/Alfred입니다
브라우저 기반 대시보드의 가장 큰 적은 다른 대시보드가 아니라 Raycast입니다. Cmd+Space → "project-a" → Enter → VSCode 열림이 이미 가능해요. 여기에 이기려면, Raycast가 못 하는 것 — 서버 상태 모니터링, git 변경사항 일괄 확인, 크로스 프로젝트 검색, Morning Routine — 을 해야 합니다. "프로젝트를 빨리 여는 도구"로는 Raycast를 못 이겨요. **"프로젝트의 상태를 한눈에 파악하는 도구"**로 포지셔닝해야 합니다.
가장 중요한 한 가지
지금 기능이 너무 많습니다. 22개 이상의 완성된 기능이 있는데, 그중 매일 쓸 건 4~5개예요. 나머지를 만드느라 쓴 시간에 서버 상태 모니터링과 Morning Routine 하나를 만들었으면, 이미 "못 끄는 앱"이 됐을 수 있습니다. 앞으로의 개발은 "추가"가 아니라 **"깊이"**에 투자하세요. 기능 10개를 반쯤 만드는 것보다, 핵심 3개를 완벽하게 만드는 게 리텐션을 만듭니다.
